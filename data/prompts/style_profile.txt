If you include topic names, product names, tool names, industry labels, or niche-specific references, the downstream prompts will propagate stale content and trap the creator in their past work. Write the profile as if the creator were about to pivot into a completely different subject area — every observation must still apply.

**CRITICAL PRINCIPLE 2 — SEPARATE PROVEN PATTERNS FROM ANTI-PATTERNS.**
Not all patterns in this creator's history are worth preserving. Some are proven winners (they correlate with high engagement). Some are lazy defaults or structural flaws (they correlate with poor engagement). You must separate the two and explicitly name both — so downstream tools know what to reproduce AND what to actively avoid. Quote literal examples of each so the downstream LLM can pattern-match against concrete text, not just abstract labels.

## Required Output Structure

Write the profile in clean markdown. Begin with the header `# Style Profile: @{username}`. Use exactly the three sections below — no other sections, no introduction, no conclusion.

### 1. Voice & Personality

Extract **unconditionally from all transcripts** — this section is performance-independent. It captures who the creator IS, not what works for them.

Cover:
- **Tone**: energy level, register (conversational / formal / hushed / raw), emotional range
- **Cadence**: typical WPM, variance pattern (when do they speed up / slow down), pause style
- **Vocabulary register**: jargon level, sentence length, sentence complexity
- **Colloquialisms & regionalisms**: British / American / other; specific idioms they actually use
- **Signature phrases**: short recurring phrases that are stylistic (e.g. "I'm giving it all away for free" is stylistic; "Make.com is rubbish" is topical and MUST be excluded)
- **Self-presentation**: how they frame themselves relative to the audience (peer / authority / underdog / insider / tired-parent-building-in-trenches / etc.)
- **Humour style**: self-deprecating / dry / sarcastic / deadpan / absent

Every claim must be backed by a short quoted example from a transcript, cited as `[video_id]`.

**FORBIDDEN IN THIS SECTION**: naming any tools, products, companies, industries, topic categories, or current events. If you find yourself writing "he talks a lot about X", rewrite it as a structural observation without X.

### 2. Proven Patterns

Derived from the **top quartile** of score cards. A pattern qualifies for inclusion if EITHER (a) it appears in multiple top-quartile videos, OR (b) the playbook explicitly endorses it AND at least one top-quartile video uses it.

Cover at minimum these areas (omit any area where no clear pattern emerges): hook mechanics, opening structure, content-structure framework, pacing patterns, CTA mechanisms, closing patterns.

For each pattern, use this exact format:

```
**Pattern: `pattern_name`**
Description: <one sentence, topic-agnostic, mechanical>
Evidence:
  - "<quoted line under 25 words>" [video_id] (aggregate: <score>, <dimension>: <dimension_score>)
  - "<quoted line under 25 words>" [video_id] (aggregate: <score>, <dimension>: <dimension_score>)
Rule for downstream use: <unambiguous instruction a scriptwriting prompt should follow to reproduce this pattern>
```

Use `code_style` backticks around pattern names so they're machine-parseable.

### 3. Anti-Patterns to Avoid

Derived from the **bottom quartile** of score cards AND the creator's self-declared lazy defaults AND any pattern that violates the playbook. These are structural habits downstream prompts must NEVER reproduce.

For each anti-pattern, use this exact format:

```
**Anti-pattern: `pattern_name`**
Examples from the creator's own content:
  - "<quoted opening or line under 25 words>" [video_id] (aggregate: <score>)
  - "<quoted opening or line under 25 words>" [video_id] (aggregate: <score>)
Why it fails: <one or two sentences — cognitive, algorithmic, or audience-behaviour mechanism>
Explicit rule: <unambiguous negative instruction for downstream prompts>
```

Include self-declared lazy defaults even if no bottom-quartile evidence supports them — cite the source as `[self-declared]` instead of a video_id.
